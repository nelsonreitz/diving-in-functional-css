# Introduction

- Utility-first CSS (a.k.a Functional CSS or Atomic CSS)
- Different approach, ditch semantic classes and old "CSS Zen Garden" mentality
- Mention frameworks

- Utility classes like .mb-2, .text-center, .flex

</TwoCols>

<Notes>

- Se détache des classes semantic
- Mentionner familiarités avec Bootstrap
- Résumer aux utility classes comme `.mb-2`, etc.
- Mentionner Frameworks (Tailwind, Tachyons, Basecss)

</Notes>

---

<TwoCols firstCol="33%" secondCol="66%">

# Problems

Why? What is it trying to solve?

- Prevents over abstraction
- You can't predict the future
- Scaling and maintaining

</TwoCols>

<Notes>

- Exemple réécrire du CSS sur un vieux projet
- La cascade finit toujours par foirer
- Se vérifie complètement dans mon cas de figure, arrivée

</Notes>

---

<TwoCols firstCol="33%" secondCol="66%">

# Solutions

How does it solve the problems?

- Provide "toolbox" of utilities
- You only write classes, no CSS
- Can extend to create components (useful for buttons and atomic elements)

</TwoCols>

<Notes>

- Repense la façon de lier le CSS
- Ne plus écrire de CSS

</Notes>

# Pros

- Constrain design system
- Easy handover / teamwork
- Framework agnostic
- Quick design in browser

</TwoCols>

<Notes>

- Contributeurs obligés de rester dans les règles du design system

</Notes>

<TwoCols firstCol="33%" secondCol="66%">

# Cons

- Themes (`.text-black` need to change to `.text-white` on dark theme)
- Web Inspector experience
- Accessibility classNames
- Old-fashioned integration (copypaste markup) can get difficult => Lead to next slide

</TwoCols>

<Notes>

- Implémentation irréaliste avec workflow copypaste du markup actuel
